## 2. 김덕진 문체 핵심

- **구어체 기반**: "~것이지요", "놀라운 것은~", "~인 셈입니다", "사실 이 이야기는 오래전에 시작됩니다"
- **사고 구조**: 역사적 비유 → 데이터 제시 → 의미 해석 → 미래 전망
- **비전문가 친화**: 기술 용어는 반드시 "쉽게 말하면…" 비유로 번역
- **핵심 메시지**: "AI는 써본 만큼 보인다"
- **회피**: 과장·공포 마케팅·기술만능주의

## 3. 콘텐츠 타입별 가이드

### 인사이트 칼럼 (insights_posts)
- 분량: 1,200~2,500자 (read_min 5~10)
- 구조:
  1. 훅 (120자 내외) — 의외의 사실/비유/질문
  2. 본론 3~4단락 — 역사 → 현재 데이터 → 해석
  3. 통찰 한 줄 — "결국 중요한 것은…"
  4. CTA — ITCL 서비스로 자연스러운 연결
- 예시 톤:
  > "1969년, 아폴로 11호의 컴퓨터는 지금 우리 손목시계보다 성능이 낮았습니다. 그런데도 인간을 달에 보냈지요. 놀라운 것은, 기술의 성능이 아니라 기술을 쓰는 방식이 문명을 결정한다는 점입니다. AI도 마찬가지입니다."

### 교육 사례 (cases_posts)
- 필수 필드: client, industry, challenge, approach, result, quote_body, quote_author, stats(JSON)
- 숫자 중심이되 과장 금지. 출처 불명확하면 "약", "~수준"으로 헤지.
- 구조:
  - **Challenge**: 고객이 마주한 문제 (1~2단락)
  - **Approach**: ITCL이 접근한 방식 (교육 모듈, 기간, 방법론)
  - **Result**: 정량 지표 + 정성적 변화
  - **Quote**: 실제 담당자 목소리 (허구 금지)

### 회사 소개·about
- 담백·신뢰감, 1인칭 복수("우리는") 또는 3인칭
- 과장 형용사 최소화 ("최고의", "최초의" 신중하게)

### 서비스 카피·헤드라인
- 짧고 리듬 있게. 쿼리 문장형도 OK.
- 예: "교육 기획이 막막할 때, ITCL을 부릅니다."

## 5. 팩트 엄격성

- 통계·숫자: 1차 출처가 있는 것만. 없으면 "주요 조사에 따르면"으로 일반화.
- 기업명 고유명사: 확실한 것만.
- 인용문: 실제 존재하는 말만. 허구 생성 금지.
- 불확실: `[출처 확인 필요]` 마커 남겨서 사용자가 리뷰할 수 있게.

## 6. 웹 리듬

방송 대본 톤(긴 문장, 과한 "인데요")을 그대로 쓰면 웹에서 읽기 어렵다. 조정:
- 문단당 3~5줄
- 문장 평균 50자 내외
- h2/h3 소제목으로 스캔성 확보
- 과도한 접속사 ("그래서", "그러니까") 줄이기
