[턴 34] 깨짐 패턴 분석 — 결정적 진단
================================================================================

클로드 (문제 해결 핵심 순간):
- 확대 캡처로 패턴 발견:
  · ✅ 하드코딩 한글 ("새 문의가 접수되었습니다", "주제", "담당자", "ITCL 연구소")

  · ❌ 동적 한글 (data.topic, data.name, data.company 등 사용자 제출값)
    → "���� �P�" 같은 mojibake로 깨짐
- 원인 확정: JSON.stringify + fetch()에서 Content-Type에 charset=utf-8 없음
  → 중간 HTTP 계층이 UTF-8 바이트를 Latin-1로 해석 → 원래 한글 바이트 손상
  → 하드코딩된 문자열은 JS 런타임이 타이트하게 잡아서 괜찮지만,
     req.body를 통해 들어온 동적 문자열은 이 경로에서 깨짐

- sendEmail() 함수 재작성:
  ```js
  const body = Buffer.from(JSON.stringify(payload), 'utf-8');  // 바이트 고정
  fetch('...', {
    headers: {
      'Content-Type': 'application/json; charset=utf-8',       // 명시적 charset
      'Content-Length': body.length.toString(),                // 길이 정확히

    body,
  });
  ```

[Part 2 총평 — 강의 메시지]
================================================================================

이 세션에서 Claude Code가 보여준 것:

1. [우선순위 재조정] 덕진님이 도메인(2번)보다 이메일(3번)을 먼저 하자고 했을 때
   의존성 점검 → "도메인 없이도 이메일 가능(기본 발신주소 활용 후 나중에 업그레이드)"
   즉시 판단하고 진행.

2. [부분 실패 진단] 첫 테스트에서 Gmail만 도착하고 itcl.kr 안 왔을 때
   즉시 Resend 무료 플랜의 "도메인 미인증 시 가입 이메일만 허용" 정책 파악
   → 해결책(도메인 인증) 제시.

3. [PowerShell 스크린 + 클립보드 조합] Resend UI가 값을 truncate 했을 때
   "덕진님이 클릭 → Get-Clipboard로 전체 값 획득" 이라는 새로운 패턴 발명.
   CMD 복붙 이슈를 가진 사용자에게 TXT 파일로 모든 레코드 완성 제공.

4. [도메인 작업 안전성] 기존 Google Workspace MX를 절대 건드리지 않고
   추가 서브도메인(send.itcl.kr, resend._domainkey.itcl.kr)으로만 작업하는
   설계 — 실수로 메일 수신 중단되는 일 방지.

5. [mojibake 진단] 한글 깨짐 현상에서 "하드코딩 vs 동적" 비교만으로 원인 특정.
   Content-Type charset 누락 → Latin-1 오해석 → UTF-8 재인코딩 손실이라는
   미묘한 인코딩 chain 3줄 수정으로 해결.

6. [브라우저 파일 보안] 모든 시크릿 (sb_secret_..., RESEND_API_KEY)은
   채팅 절대 안 지나가고 Vercel env var에 브라우저↔브라우저로만 이동.

핵심 메시지: 복잡한 인프라(이메일 서비스 + DNS + 인증 + 인코딩)를 사용자의
제약(CMD 복붙 불가, 멀티모니터, 도메인 미결정)에 맞춰 설계하고, 예상 못한
이슈(부분 성공, mojibake)를 진단+수정까지 주도. AI가 "도구"가 아닌
"시니어 엔지니어 역할"을 실제로 해냄.
